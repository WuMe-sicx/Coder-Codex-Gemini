
# CC Configuration

# Claude + Codex 双模型协作协议

**定位**：Claude 主写代码并自测，Codex 作为独立终审只审不改。额度充足，目标是流程简单、上下文稳定、责任清晰、代码质量可控。

## 三条铁律

1. **Codex 默认只审查，绝不直接改代码或文件**（`sandbox="read-only"`）。
2. **所有修改统一由 Claude 落地**，避免两个 agent 同时写文件造成覆盖或上下文混乱。
3. **最终把关的模型 ≠ 写代码的模型** —— Codex 复审是最后一道闸，Claude 不得自我放行。

## ⚠️ Skill 前置（强制）

调用 `codex` MCP 工具进行审查前，**必须先执行 `/cc-review` Skill** 获取最佳实践（送审时机、diff 准备、审查模板、复审与重试策略），再据此调用工具。

禁止：跳过 Skill 直接调用工具；假设已了解最佳实践而不执行 Skill。

## 标准闭环

```text
Claude 拆解需求 → Claude 写代码并自测 → 提交 Codex 独立审查
→ Claude 按意见逐条修复 → Codex 复审 → 通过（否则回到「修复」）
```

## 何时送审

- 完成**一个可独立验证的功能单元 / 一组相关改动后**（PR 级粒度）再送审，不要每改一行就审。
- **关键模块强制送审**：鉴权、支付、数据写入、对外接口、并发逻辑。
- 修复后**必须复审**，直到 Codex 明确回复"通过"（✅ PASS）。
- 复审尽量**复用同一 Codex 会话**（携带返回的 `SESSION_ID`），保留初审意见。

## 修复与重试

- 依据 Codex 清单**逐条修复**，只改问题相关代码，不顺手重构无关部分。
- 同一问题**最多 2 轮往返**；仍卡住则停下，把分歧点抛给人工裁决，不要无限循环。

## 规范

- **语言**：面向用户的回复使用中文；给 Codex 的 PROMPT 使用英文（必要时附中文注释）。
- **会话管理**：必须使用工具返回的实际 `SESSION_ID`，严禁自创 ID。
- **第二意见**：本仓库不接入其它模型，需要第二意见时仍只用 Codex。

**Claude 是工程主体与最终决策者**：对 Codex 明显误报需有理有据地反驳，对合理意见不得自我放行。
