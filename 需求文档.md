# CLAUDE.md — 协作规则:Claude 写,Codex 审

## 定位

精简为 **Claude + Codex 双模型协作**。额度充足,目标不再是省钱,而是:流程简单、上下文稳定、责任清晰、代码质量可控。

- **主写 = Claude(本 agent,Max20)**:需求分析、任务拆解、代码实现与修改、自测、按审查意见修复。
- **终审 = Codex(Pro×5)**:独立 Code Review,只审不改。

已移除:Coder、Gemini、OpenCode / Sisyphus,以及"便宜模型干活、贵模型调度"的全部成本优化叙事与多模型流水线配置。

## 三条铁律

1. **Codex 默认只审查,绝不直接改代码或文件。**
2. **所有修改统一由 Claude 落地**,避免两个 agent 同时写文件造成覆盖或上下文混乱。
3. **最终把关的模型 ≠ 写代码的模型** —— 这里 Codex 复审是最后一道闸,Claude 不得自我放行。

## 标准闭环

```text
Claude 拆解需求
→ Claude 写代码并自测
→ 提交 Codex 独立审查
→ Claude 按意见修复
→ Codex 复审
→ 通过(否则回到「修复」)
```

## 何时调用 Codex 审查

- 完成**一个可独立验证的功能单元 / 一组相关改动后**(PR 级粒度)再送审,不要每改一行就审。
- **关键模块强制送审**:鉴权、支付、数据写入、对外接口、并发逻辑。
- 修复后**必须复审**,直到 Codex 明确回复"通过"。
- 复审尽量**复用同一 Codex 会话/上下文**,让它保留初审意见,复审更连贯。

## 给 Codex 的审查指令模板

> 请只做独立代码审查,**不要修改代码**。逐项检查并指出「问题 / 位置 / 严重级别 / 建议」:
>
> 1. **逻辑正确性**:是否实现既定需求,有无逻辑漏洞。
> 2. **边界条件**:空值、越界、异常输入、并发竞态。
> 3. **安全风险**:注入、越权、敏感信息泄露、不安全依赖。
> 4. **测试缺口**:关键路径与边界是否有覆盖,断言是否有效。
> 5. **可维护性/性能**:有无明显问题(可选)。
>
> 无问题则明确回复"通过"。

## 修复与重试策略

- Claude 依据 Codex 清单**逐条修复**,只改问题相关代码,不顺手重构无关部分。
- 同一问题**最多 2 轮往返**;若仍卡住,停下并把分歧点抛给人工裁决,**不要无限循环**。

## 调用方式

Codex 作为已配置的 review 工具调用(codex CLI / MCP)。本仓库**不接入其它模型**;需要第二意见时仍只用 Codex,不默认引入 Gemini。
