# Codex 终审协作流程

## 定位

**Claude + Codex 双模型协作**。目标不是省钱，而是：流程简单、上下文稳定、责任清晰、代码质量可控。

- **主写 = Claude（本 agent）**：需求分析、任务拆解、代码实现与修改、自测、按审查意见修复。
- **终审 = Codex**：独立 Code Review，**只审不改**。

## 三条铁律

1. **Codex 默认只审查，绝不直接改代码或文件**（`sandbox="read-only"`）。
2. **所有修改统一由 Claude 落地**，避免两个 agent 同时写文件造成覆盖或上下文混乱。
3. **最终把关的模型 ≠ 写代码的模型** —— Codex 复审是最后一道闸，Claude 不得自我放行。

## 标准闭环

```text
Claude 拆解需求
→ Claude 写代码并自测
→ 提交 Codex 独立审查
→ Claude 按意见逐条修复
→ Codex 复审
→ 通过（否则回到「修复」）
```

## 何时调用 Codex 审查

- 完成**一个可独立验证的功能单元 / 一组相关改动后**（PR 级粒度）再送审，不要每改一行就审。
- **关键模块强制送审**：鉴权、支付、数据写入、对外接口、并发逻辑。
- 修复后**必须复审**，直到 Codex 明确回复"通过"（✅ PASS）。
- 复审尽量**复用同一 Codex 会话**（携带上一轮返回的 `SESSION_ID`），让它保留初审意见，复审更连贯。

## 送审前准备（重要）

调用 Codex 前，Claude 先取 `git diff` 并附在 PROMPT 中，让 Codex 精准审查变更而非自行探索文件，省 token 且更准：

```bash
git diff --no-color
```

把 diff 连同「改动文件 / 改动目的」嵌入 PROMPT。详见 [codex-guide.md](codex-guide.md)。

## 给 Codex 的审查指令模板

> 请只做独立代码审查，**不要修改代码**。逐项检查并指出「问题 / 位置 / 严重级别 / 建议」：
>
> 1. **逻辑正确性**：是否实现既定需求，有无逻辑漏洞。
> 2. **边界条件**：空值、越界、异常输入、并发竞态。
> 3. **安全风险**：注入、越权、敏感信息泄露、不安全依赖。
> 4. **测试缺口**：关键路径与边界是否有覆盖，断言是否有效。
> 5. **可维护性/性能**：有无明显问题（可选）。
>
> 无问题则明确回复"通过"。

（Codex 工具内置了等价的英文 system prompt，模板用于在 PROMPT 中补充本次审查重点。）

## 修复与重试策略

- Claude 依据 Codex 清单**逐条修复**，只改问题相关代码，**不顺手重构无关部分**。
- 同一问题**最多 2 轮往返**；若仍卡住，停下并把分歧点抛给人工裁决，**不要无限循环**。

## 结论处理

| Codex 结论 | 处理 |
|-----------|------|
| ✅ PASS | 通过，进入下一步 |
| ⚠️ OPTIMIZE | 可合入；评估建议，必要时修复后继续 |
| ❌ CHANGE | 必须修复 → Claude 逐条改 → 复用会话复审 → 循环直至 PASS |

## 常见工作流程（Workflows）

> 落地动作统一记法：`写码自测 → git diff → codex 审 → 修复 → 复审`。

**A. 新功能 / 一组改动**
1. 拆需求、列影响文件与验收标准。
2. Claude 写代码 + 自测（跑通/跑测试）。
3. 完成一个可独立验证单元后：`git diff --no-color` → 嵌入 PROMPT → 调 `codex`（新会话）。
4. ❌/⚠️ → 逐条修复 → 携带返回的 `SESSION_ID` 复审 → 直至 ✅ PASS。

**B. Bug 修复**
1. 定位根因（必要时先读相关代码，别急着改）。
2. Claude 写修复 + **补回归测试** → 自测。
3. 送审，重点请 Codex 看：根因是否真消除、有无引入新边界问题、断言是否有效。
4. 复审至 PASS。

**C. 关键模块（鉴权/支付/数据写入/对外接口/并发）**
- **强制送审**，即使改动很小，Claude 不得自我放行。
- PROMPT 中显式标注高风险面，请 Codex 重点核查不可逆 / 对外可见行为。
- 必须复审到 ✅ PASS 才合入。

**D. 重构**
- 行为不变是前提：先确保有可对照的测试或冒烟验证。
- 送审时说明「纯重构、行为应保持不变」，请 Codex 核对是否有语义漂移。
- 只改目标范围，不顺手动无关代码。

**E. Codex 给了 ❌/⚠️ 但你不认同**
- 不盲从：用证据（代码/测试/规格）有理有据反驳，必要时在复审 PROMPT 中说明分歧。
- 同一问题**最多 2 轮往返**；仍僵持则停下，把分歧点抛给人工裁决，不要无限循环。

**F. Codex 工具不可用（未登录 / 未安装 / 超时）**
- `command_not_found` / `auth_required` → 提示用户 `codex login` 或安装 Codex CLI，不重试。
- 其它错误（超时/网络）→ 工具默认重试 1 次；仍失败则改用 Git + Markdown 在 `docs/`/`discuss/` 记录过程，待工具恢复后补审。

## 独立决策

Codex 的意见是最后一道闸，但 Claude 仍是工程主体：对明显误报需有理有据地反驳，而非盲从；对合理意见不得自我放行。需要第二意见时仍只用 Codex，**本仓库不接入其它模型**。
