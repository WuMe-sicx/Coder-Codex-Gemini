# Codex 终审协作流程

## 定位

**Claude + Codex 双模型协作**。目标不是省钱，而是：流程简单、上下文稳定、责任清晰、代码质量可控。

- **主写 = Claude（本 agent）**：需求分析、任务拆解、代码实现与修改、自测、按审查意见修复。
- **终审 = Codex**：独立 Code Review，**只审不改**。

## 核心准则：Karpathy 工程准则（每次必守）

> 写码 / 修复 / 重构前**先过一遍**这套准则（可执行 `/karpathy-guidelines` 取完整版）；
> Codex 端**已内置同一准则**到 system prompt，会按同一标准审查 —— 两边对齐。

1. **想清再写**：显式说明假设；有歧义先提出别默默选；有更简方案就直说。
2. **简单优先**：只写解决问题的最小代码；不加未要求的特性 / 抽象 / 配置 / 不可能分支的错误处理。
3. **外科手术式改动**：每行改动都能追溯到需求；不顺手重构没坏的代码、不改无关格式；只清理**自己**产生的孤儿，不动既有死代码（除非被要求）。
4. **目标驱动**：把任务转成可验证目标（先写复现/验收检查再使其通过），循环到验证通过。

## 三条铁律

1. **Codex 默认只审查，绝不直接改代码或文件**（`sandbox="read-only"`）。
2. **所有修改统一由 Claude 落地**，避免两个 agent 同时写文件造成覆盖或上下文混乱。
3. **最终把关的模型 ≠ 写代码的模型** —— Codex 复审是最后一道闸，Claude 不得自我放行。

## 标准闭环

```text
Claude 拆解需求
→ Claude 写代码并自测
→ 提交 Codex 独立审查
→ Claude 按意见逐条修复
→ Codex 复审
→ 通过（否则回到「修复」）
```

## 何时调用 Codex 审查

- 完成**一个可独立验证的功能单元 / 一组相关改动后**（PR 级粒度）再送审，不要每改一行就审。
- **关键模块强制送审**：鉴权、支付、数据写入、对外接口、并发逻辑。
- 修复后**必须复审**，直到 Codex 明确回复"通过"（✅ PASS）。
- 复审尽量**复用同一 Codex 会话**（携带上一轮返回的 `SESSION_ID`），让它保留初审意见，复审更连贯。

## 送审前准备（重要）

调用 Codex 前，Claude 先取 `git diff` 并附在 PROMPT 中，让 Codex 精准审查变更而非自行探索文件，省 token 且更准：

```bash
git diff --no-color
```

把 diff 连同「改动文件 / 改动目的」嵌入 PROMPT。详见 [codex-guide.md](codex-guide.md)。

## 给 Codex 的审查指令

`codex` 工具已内置等价的英文 system prompt（审查清单：逻辑正确性 / 边界条件 / 安全风险 / 测试缺口 / 可维护性，并要求以 ✅/⚠️/❌ 结尾），**PROMPT 里不必重列清单**，只需嵌入 diff 并补「改动文件 / 目的 / 本次重点」。唯一送审模板见 [codex-guide.md](codex-guide.md)。

## 修复与重试策略

- Claude 依据 Codex 清单**逐条修复**，只改问题相关代码，**不顺手重构无关部分**。
- 同一问题**最多 2 轮往返**；若仍卡住，停下并把分歧点抛给人工裁决，**不要无限循环**。

## 结论处理

| Codex 结论 | 处理 |
|-----------|------|
| ✅ PASS | 通过，进入下一步 |
| ⚠️ OPTIMIZE | 可合入；评估建议，必要时修复后继续 |
| ❌ CHANGE | 必须修复 → Claude 逐条改 → 复用会话复审 → 循环直至 PASS |

## 常见工作流程（Workflows）

各场景都走同一闭环 `写码自测 → git diff → codex 审 → 修复 → 复审`，下面只列**该场景的额外动作**：

| 场景 | 额外动作 |
|------|---------|
| **A. 新功能 / 一组改动** | 先拆需求列影响文件与验收标准；完成一个可独立验证单元后再送审（新会话）。 |
| **B. Bug 修复** | 先定位根因别急着改；修复时**补回归测试**；送审请 Codex 重点看根因是否真消除、有无新边界问题、断言是否有效。 |
| **C. 关键模块**（鉴权/支付/数据写入/对外接口/并发） | **强制送审**（即使改动很小，不得自我放行）；PROMPT 显式标注高风险面，请重点核查不可逆 / 对外可见行为；必须复审到 ✅ PASS 才合入。 |
| **D. 重构** | 前提是行为不变：先备好可对照测试或冒烟；送审说明「纯重构、行为应保持不变」请核对语义漂移；只改目标范围。 |
| **E. 不认同 Codex 的 ❌/⚠️** | 不盲从：用代码/测试/规格证据反驳，在复审 PROMPT 中说明分歧；同一问题最多 2 轮往返，僵持则抛人工裁决。 |
| **F. 工具不可用**（未登录/未安装/超时） | `command_not_found` / `auth_required` → 提示 `codex login` 或安装，不重试；其它错误工具默认重试 1 次，仍失败则用 Git + Markdown 在 `docs/`/`discuss/` 记录，待恢复补审。 |

## 独立决策

Codex 的意见是最后一道闸，但 Claude 仍是工程主体：对明显误报需有理有据地反驳，而非盲从；对合理意见不得自我放行。需要第二意见时仍只用 Codex，**本仓库不接入其它模型**。
